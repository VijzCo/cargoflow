# CargoFlow — Chunk 5: Packing Lists & PDF Export

## What's new

- **Packing Lists page** at `/packing-lists` — full list of all generated packing lists with search, PDF download per row, and delete.
- **"Generate packing lists" button on vessel detail page** — opens a preview-then-commit dialog showing exactly what will be created.
- **One packing list per (container, supplier)** — a container with items from 3 suppliers generates 3 separate packing lists. Format: `PL-{VesselID}-{ContainerNumber}-{SupplierIdPrefix}`.
- **Professional PDF generation** — client-side using jsPDF + jspdf-autotable. Navy header, two-column meta block (PL #, vessel, container, destination, ETD/ETA), items table grouped by PO with totals row, weight summary grid, signature lines, and footer with generation timestamp.
- **Packing list detail page** at `/packing-lists/[plId]` — header stats, supplier info, items table, one-click PDF download.
- **Idempotent generation** — if you click "Generate" twice, the second time skips existing packing lists. Optional "Replace existing" checkbox lets you regenerate with fresh data.
- **Item backlinks** — each item gets a `packingListId` field when added to a packing list. Lets you trace an item from PO to its packing list.
- **Activity log entries** for every generation and deletion.

## End-to-end test flow

This assumes you've completed Chunk 4 (i.e. you have at least one vessel with sealed containers, OR you can generate packing lists at any point before dispatch).

### Step 1 — Get to a vessel with containers

Either reuse a vessel from your Chunk 4 testing, or:
1. Have one or more items at `Loaded` status (assigned to a container).
2. Seal that container.
3. Create a vessel and attach the container.

You don't need to dispatch — packing lists can be generated at any time before dispatch.

### Step 2 — Generate

1. **Vessels → click your vessel** → vessel detail page.
2. Click **"Generate packing lists"** (next to "Attach containers" and "Dispatch vessel").
3. **Preview screen** shows you a row for each (container × supplier) combo:
   - Container number
   - Supplier name
   - Item count
   - Total CBM
   - The expected packing list number (e.g. `PL-CG-2026-04-MSCU1234567-ABC123`)
   - Status: green "New" badge or amber "Exists" badge
4. Click **"Generate N"** at the bottom.
5. Success toast → you're redirected to `/packing-lists`.

### Step 3 — Download a PDF

1. On the packing lists page, find your new packing list.
2. Click **"PDF"** in the actions column → PDF generates in your browser and downloads.
3. Open it. You should see:
   - Supplier letterhead at top-left, "PACKING LIST" header at top-right
   - Meta block: PL number, container, vessel, destination, ETD, ETA
   - Items table grouped by PO with a slate-grey row separating PO groups
   - Bottom row in dark navy: TOTAL with quantity, packages, CBM, gross weight
   - Six-card weight summary grid below the table
   - Signature lines: ISSUED BY / APPROVED BY
   - Footer with "Generated by CargoFlow" + date

### Step 4 — Re-generate (idempotency test)

1. Go back to the vessel detail page.
2. Click "Generate packing lists" again.
3. Preview now shows amber "Exists" badges for the ones you just made.
4. Click "Generate" (the button shows 0 because everything exists).
5. Toast confirms 0 created.
6. Now tick **"Replace existing packing lists with fresh data"** at the bottom.
7. Click "Generate" — this time the button enables and replaces them. Use this if you've updated item CBMs/weights and want the latest data in the PDF.

### Step 5 — Detail page

1. Click any packing list number → detail page.
2. See header stats, supplier letterhead, KPI grid, full item table.
3. Click "Download PDF" at the top-right.

### Step 6 — Delete

1. From the list, click the trash icon next to any packing list.
2. Confirm → it's deleted, items have their `packingListId` backlink cleared.

## How the PDF layout maps to your sample

Your `sample_packing_list.xls` had:
- Supplier name + address as a letterhead at the top → **PDF top-left**
- "PACKING LIST" title → **PDF top-right**
- Invoice/PL number, ETD, FROM, TO → **PDF meta block**
- Item table with style, color, qty, packages, CBM, weight → **PDF main table**
- Totals row in dark color → **PDF dark navy totals row**
- Weight breakdown (gross/net) → **PDF six-card summary grid**

The visual style is intentionally more modern than the legacy Excel format (proper typography, color hierarchy, less crowded), but the data fields and ordering match what shipping carriers and customs offices expect.

## How idempotency works

Each packing list has a deterministic number based on `vesselId + containerNumber + supplierIdPrefix`. The same (vessel, container, supplier) tuple always produces the same PL number. On generation:

- If a PL with that number doesn't exist → create it.
- If it exists and `regenerate` is false → skip.
- If it exists and `regenerate` is true → delete the old one + create a fresh copy.

This means you can safely click "Generate" before each PDF download to ensure the data is current — or you can generate once and just re-download the PDF (the PDF reads current data from the items at download time, so it's always fresh anyway).

## Cost & performance notes

**The PDF is generated in the browser.** No Cloud Function, no Puppeteer, no server-side rendering — jsPDF builds the file entirely in your browser using JavaScript. This means:
- Zero Firebase Functions cost.
- No file storage cost (the PDF lives in your downloads folder, not in Firebase Storage).
- Sub-second generation for typical packing lists (~50 items).
- The downside: the PDF lives only in the user's downloads folder. If you want a permanent archive, save the PDFs manually somewhere or we can wire up Firebase Storage in a future chunk.

**The packing list document in Firestore** stores only metadata: PL number, vessel/container/supplier refs, totals, item IDs. The actual PDF is rebuilt from these refs + item data each time you click download.

## Known limitations

- **PDF doesn't include a company logo.** The settings doc has a `logoUrl` field reserved for this — we'll wire it in when we build the settings UI in Chunk 6.
- **PDF only renders the first page.** With more than ~30 items per packing list, the items table will spill onto page 2 but the header won't repeat. jsPDF-autotable can be configured to repeat headers; happy to enable that if you need it (typical packing lists from your sample fit on one page).
- **No bulk PDF download** — each PDF requires its own click. If you generate 10 packing lists for one vessel, you'd click "PDF" 10 times. We can add a "Download all as ZIP" in a future chunk if needed.
- **No email-this-PDF** — saving to disk is the only option for now.
